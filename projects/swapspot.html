Swapspot

A fast visual-memory game built around one escalating challenge: study a grid of icons, spot the tiles that changed position, and hold a streak long enough for the board to become genuinely difficult.

Watch. Swap. Find. Repeat.

Swapspot makes its rule legible immediately, then earns its difficulty through timing and scale. A fresh grid is shown to the player, selected icons trade places, and the player has to name the changed tiles before their lives run out.

Preparation

Memorize the grid

Every round begins with a short reading window. The board is not random noise—the player gets a real chance to build a visual map.

Challenge

Spot the swap

Once the board changes, the game asks for the exact number of affected tiles and prevents duplicate guesses.

Resolution

Earn the next round

Correct choices raise score and streak; misses reset momentum, cost a life, and eventually send the run to a game-over screen.

Difficulty is a changing board, not a vague label

The six tiers are defined in code. They grow the board, adjust round reward, and increase how many changes must be found. That makes a level-up feel like a new test rather than a new badge.

Every state has its own feedback

A memory game needs the player to understand why a round succeeded or failed without reading a manual. Swapspot treats feedback as a system: state, visuals, sound, overlays, and settings all communicate the current moment of play.

Round state

Menu, preparing, swapped, round result, and game-over states make the lifecycle explicit and stop input being accepted at the wrong moment.

Audio

Click, swap, pop, correct, wrong, level-up, and game-over sounds create an immediate response while respecting user-set music and SFX volume.

Presentation

Correct and wrong reveals, a level-up overlay, responsive grid sizing, dark mode, and settings preserve clarity as the board becomes dense.

The swap algorithm preserves identity

The player is not guessing which positions changed abstractly. Each tile owns a stable id plus an icon name and pastel colour. A round selects unique indices, rotates their tile objects through a cycle, and records the ids of every affected item. Guess verification compares two sets, so order does not matter but exact membership does.

Generate

size × size itemsStable id, icon key, pastel colour
Shuffled icon poolMinimises duplicates on smaller boards

Transform

Unique random indicesTwo on Easy–Expert; three on Master–Legend
Cyclic rotationEvery selected position changes without losing objects

Verify

Fixed selection countRound resolves as soon as required guesses are chosen
Set equalityExact ids, independent of click order

Five states control every interaction

The root component owns a small state machine and a single timer ref. That ensures the board is clickable only during the challenge window and that delayed transitions can be cleared if the component unmounts.

Scoring rewards both survival and consistency

A correct round earns the tier’s base points, ten points for each current round, and a streak bonus of 50 points per existing streak step. Five consecutive correct rounds reset the streak and advance one tier until Legend. A wrong answer removes one of three lives and clears the streak.

ComponentFormula / behaviour
Tier base100, 150, 250, 400, 600, or 1,000 points.
Round growthCurrent round × 10.
Streak growthExisting streak × 50.
Level thresholdFive correct answers in sequence.
Failure costOne life and streak reset; third miss ends the run.
Memory windowStarts at two seconds; later rounds reduce toward an 800 ms floor by 50 ms per round.

A leaderboard that works offline and upgrades online

Swapspot’s storage layer first checks whether Supabase environment variables are configured. If they are, scores are read and inserted against a global leaderboard table. If not—or if a request fails—the same interface falls back to browser localStorage.

Top-five rule

A score qualifies automatically while fewer than five entries exist; after that it must beat the current lowest score.

Safe fallback

Network or configuration failure returns local results instead of breaking the menu or game-over screen.

Cross-component refresh

A custom leaderboardUpdate event refreshes the menu after submission or a user-confirmed data clear.

Global indicator

The menu knows whether entries are global or device-local and presents the source to the player.

Submission state

The game-over component checks asynchronously, validates a name, shows save progress, then locks the submitted result.

Stored context

Each row includes name, score, difficulty reached, and date—not just a number.

The soundtrack and effects are synthesized live

The 255-line sound manager uses the Web Audio API rather than imported music files. It creates a shared context and gain graph, then generates tones and noise with oscillators, filters, timed envelopes, and repeating musical patterns.

The grid is constrained by the smaller viewport dimension

The game board uses min(90vw, 65vh) and a square aspect ratio. That keeps a 10×10 grid inside short laptop screens as well as narrow phones. Column classes are chosen from the actual board side, gaps shrink on small screens, and the surrounding layout reserves room for the header rather than forcing vertical scrolling mid-round.

Memoized tiles

GridItem is wrapped in React memo so unchanged tiles avoid unnecessary rerenders.

Input lock

Tiles receive disabled outside SWAPPED state, preventing accidental guesses during memory and reveal phases.

Dark mode

Settings toggle a root class, allowing the same component tree to switch the full Tailwind palette.

React boundaries

The UI is split around game responsibilities

App.tsx owns state and timers; presentation and persistence sit in smaller components and utilities. Round transitions therefore stay separate from menus, modals and score submission.

The 353-line coordinator owns grid, difficulty, statistics, selection, timers and transitions. A timer ref prevents stale phase callbacks.

GridItem

A memoized tile renders selected, correct, wrong, disabled and moving states. Stable IDs preserve identity as objects move.

Header

Score, round, streak, lives and transient messages form one always-visible game status.

Menu

The 154-line opening screen presents six difficulty configurations and refreshes the leaderboard on local update events.

GameOver

The end screen checks qualification asynchronously, collects a name, submits it and exposes retry and home.

Settings

Sound preferences and local-data clearing remain outside the round machine and use the same leaderboard refresh path.

Resilient persistence

Online enhancement without an online dependency

Swapspot treats Supabase as optional. Environment variables decide whether a real client exists; without them, the complete game uses localStorage. Remote reads and writes retain a local fallback, which matters for a game expected to start instantly.

The local board uses the same entry shape, sorts by score and keeps five records. High-score detection handles both an under-filled board and the threshold set by its lowest score. Saving dispatches a custom leaderboardUpdate event so an already-mounted menu refreshes without reloading. The interface can label competition global or local while game code calls one storage API.

Product consequence: backend configuration changes the reach of competition, not whether the game works. An offline deployment still preserves scores, settings and a coherent game-over flow.

Status

Swapspot is a launch-ready React and TypeScript game with a complete five-state round loop, six difficulty tiers, responsive 3×3 through 10×10 boards, synthesized music and effects, local leaderboard persistence, optional global Supabase scores, high-score submission, settings, dark mode, and polished result feedback. Remaining work is primarily product hardening: higher-tier balance testing, touch-first QA, accessibility review, and stronger automated coverage of timers and progression.
